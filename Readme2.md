## ВВЕДЕНИЕ

Производственная практика проходила в рамках разработки информационной системы для автоматизации управления заявками абонентов ПАО «Ростелеком» — крупнейшего оператора связи в России, предоставляющего услуги фиксированной и мобильной связи, широкополосного доступа в Интернет, цифрового телевидения и облачных решений.

**Актуальность.** ПАО «Ростелеком» обслуживает миллионы абонентов по всей России. Ежедневно поступают тысячи заявок на подключение, ремонт и модернизацию телекоммуникационного оборудования. Для обработки этих заявок задействованы выездные технические специалисты, диспетчеры и руководители. В условиях большого потока обращений критически важна автоматизация процессов: от создания заявки до её закрытия с отчётом о выполненных работах.

Ручное управление заявками (бумажные журналы, телефонные звонки, электронная почта) приводит к ряду проблем:
- потеря заявок и дублирование;
- отсутствие оперативного контроля за ходом выполнения;
- невозможность отслеживания истории изменений;
- сложность формирования отчётности;
- отсутствие уведомлений для техников о новых назначениях.

**Цель работы** — разработка мобильного приложения с серверной частью для автоматизации процесса управления заявками абонентов Ростелеком. Система предназначена для выездных техников, диспетчеров и руководителей.

**Задачи практики:**
1. Анализ предметной области — изучение бизнес-процессов обработки заявок абонентов Ростелеком.
2. Проектирование базы данных — разработка ER-диаграммы, словаря данных, нормализация до 3НФ.
3. Реализация базы данных в PostgreSQL.
4. Разработка серверной части (REST API) на языке Go с использованием фреймворка Gin и ORM Ent.
5. Разработка мобильного приложения на Flutter с подключением к API.
6. Реализация ролевой модели доступа (администратор, диспетчер, техник, руководитель).
7. Тестирование системы.
8. Подготовка документации.

**Стек технологий:**
- Серверная часть: Go (Golang), Gin (HTTP-фреймворк), Ent (ORM)
- Мобильное приложение: Flutter (Dart), Provider (управление состоянием)
- База данных: PostgreSQL 16
- Аутентификация: JWT (access + refresh tokens)
- Инструменты: Git, VS Code, pgAdmin

## ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

### 1.1 Характеристика предприятия ПАО «Ростелеком»

ПАО «Ростелеком» — крупнейший в России провайдер цифровых услуг и решений, присутствующий во всех сегментах рынка и охватывающий домохозяйства, государственные и частные организации. Компания занимает лидирующие позиции на рынке услуг широкополосного доступа в Интернет и платного телевидения.

Основные услуги для физических лиц:
- **Интернет** — широкополосный доступ по технологиям GPON, FTTB, xDSL;
- **Цифровое ТВ** — интерактивное телевидение Wink, IPTV;
- **Телефония** — стационарная телефонная связь;
- **Видеонаблюдение** — облачное видеонаблюдение с мобильным приложением.

Для обслуживания абонентов Ростелеком располагает штатом выездных технических специалистов, работающих в каждом регионе присутствия. Координацию их работы осуществляют диспетчерские службы.

### 1.2 Анализ существующих бизнес-процессов обработки заявок

Типичный процесс обработки заявки абонента включает следующие этапы:

1. **Приём заявки.** Абонент обращается через контактный центр, личный кабинет или мобильное приложение. Оператор фиксирует обращение, определяет тип услуги и категорию проблемы.

2. **Диспетчеризация.** Диспетчер анализирует заявку, определяет приоритет и назначает выездного техника с учётом его загрузки и географического расположения.

3. **Выполнение работ.** Техник получает информацию о заявке, выезжает по адресу клиента, выполняет необходимые работы (подключение, ремонт, замена оборудования).

4. **Закрытие заявки.** После выполнения работ техник фиксирует результат, оставляет комментарий о проделанной работе. Заявка переходит в статус «Завершена».

5. **Контроль.** Руководитель отслеживает статистику выполнения заявок, контролирует сроки и качество работы техников.

Жизненный цикл заявки в системе:

```
Новая → Назначена → В работе → Завершена
                              → Отменена
```

### 1.3 Роли пользователей системы

В системе предусмотрены четыре роли:

| Роль | Описание | Права |
|------|----------|-------|
| **Администратор** (admin) | Системный администратор | Полный доступ, управление пользователями |
| **Диспетчер** (dispatcher) | Оператор диспетчерской службы | Создание заявок, назначение техников, контроль |
| **Выездной техник** (technician) | Технический специалист | Просмотр назначенных заявок, смена статуса, отчёт |
| **Руководитель** (supervisor) | Руководитель подразделения | Мониторинг, создание заявок, отчётность |

### 1.4 Постановка задачи

На основании анализа предметной области были сформулированы следующие требования к системе:

**Функциональные требования:**
- Авторизация пользователей с ролевой моделью доступа;
- Создание, просмотр, редактирование и удаление заявок;
- Назначение заявок на техников;
- Изменение статуса заявок с фиксацией в журнале;
- Система уведомлений о назначениях и изменениях;
- Фильтрация и поиск заявок;
- Управление пользователями (администратор);
- Экспорт данных в CSV.

**Нефункциональные требования:**
- Работа на мобильных устройствах Android;
- Время отклика API — не более 500 мс;
- Защита данных с помощью JWT-токенов;
- Поддержка одновременной работы нескольких пользователей.

## ГЛАВА 2. ПРОЕКТИРОВАНИЕ БАЗЫ ДАННЫХ

### 2.1 Концептуальная модель данных

База данных системы включает 10 таблиц, организованных по принципу нормализации до третьей нормальной формы (3НФ). Справочные таблицы вынесены отдельно для исключения избыточности данных.

Основные сущности:
- **Пользователи** (users) — сотрудники системы с ролями;
- **Заявки** (requests) — основная бизнес-сущность;
- **Адреса** (addresses) — адреса клиентов;
- **История изменений** (request_history) — журнал всех действий;
- **Уведомления** (notifications) — оповещения пользователей.

Справочники:
- **Роли** (roles) — admin, dispatcher, technician, supervisor;
- **Типы услуг** (service_types) — Интернет, Цифровое ТВ, Телефония, Видеонаблюдение;
- **Статусы заявок** (request_statuses) — new, assigned, in_progress, completed, cancelled;
- **Категории заявок** (request_categories) — Подключение, Ремонт, Консультация, Модернизация.

### 2.2 ER-диаграмма

Связи между таблицами:

```
roles (1) ──── (M) users
service_types (1) ──── (M) requests
request_statuses (1) ──── (M) requests
request_categories (1) ──── (M) requests
addresses (1) ──── (M) requests
users (1) ──── (M) requests [assigned_to — техник]
users (1) ──── (M) requests [created_by — создатель]
requests (1) ──── (M) request_history
requests (1) ──── (M) notifications
users (1) ──── (M) request_history
users (1) ──── (M) notifications
users (1) ──── (M) refresh_tokens
```

Все связи имеют тип «один ко многим». Внешние ключи обеспечивают целостность данных.

### 2.4 Нормализация до третьей нормальной формы

**Первая нормальная форма (1НФ):**
- Все поля атомарны (не содержат составных или множественных значений).
- Адрес разбит на отдельные компоненты: город, улица, дом, квартира, подъезд, этаж.
- Каждая таблица имеет первичный ключ.

**Вторая нормальная форма (2НФ):**
- Все неключевые атрибуты функционально зависят от всего первичного ключа.
- Первичные ключи — простые (автоинкрементные), поэтому частичных зависимостей нет.

**Третья нормальная форма (3НФ):**
- Транзитивные зависимости устранены.
- Справочные данные (роли, типы услуг, статусы, категории) вынесены в отдельные таблицы.
- Адреса вынесены из заявок в отдельную таблицу.
- Информация о пользователях не дублируется в заявках — используются внешние ключи.

## ГЛАВА 3. ФИЗИЧЕСКАЯ РЕАЛИЗАЦИЯ БАЗЫ ДАННЫХ

### 3.1 Выбор СУБД PostgreSQL

Для реализации базы данных выбрана СУБД PostgreSQL 16 по следующим причинам:
- Открытый исходный код, бесплатная лицензия;
- Полная поддержка стандарта SQL, транзакций ACID;
- Поддержка JSON/JSONB для гибких структур;
- Высокая производительность и надёжность;
- Совместимость с ORM Ent для Go;
- Широкое распространение в промышленных системах.

### 3.2 SQL-скрипты создания таблиц

В проекте используется двойной подход к созданию таблиц:

1. **Автоматическая миграция через Ent ORM** — при запуске Go-сервера Ent автоматически создаёт и обновляет таблицы в соответствии со схемами, определёнными в файлах `ent/schema/*.go`. Это основной способ.

2. **Справочный SQL-скрипт** (`database/schema.sql`) — содержит DDL-скрипты для ручного создания таблиц и заполнения справочников. Используется как документация и для развёртывания без Go.

Пример создания таблицы заявок (из schema.sql):

```sql
CREATE TABLE IF NOT EXISTS requests (
    id               SERIAL PRIMARY KEY,
    title            VARCHAR(300) NOT NULL,
    description      TEXT DEFAULT '',
    service_type_id  SMALLINT NOT NULL REFERENCES service_types(id),
    category_id      SMALLINT NOT NULL REFERENCES request_categories(id),
    status_id        SMALLINT NOT NULL REFERENCES request_statuses(id) DEFAULT 1,
    address_id       INTEGER REFERENCES addresses(id),
    client_name      VARCHAR(200) NOT NULL,
    client_phone     VARCHAR(20) NOT NULL,
    assigned_to      INTEGER REFERENCES users(id),
    created_by       INTEGER NOT NULL REFERENCES users(id),
    scheduled_at     TIMESTAMPTZ,
    completed_at     TIMESTAMPTZ,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_requests_status ON requests(status_id);
CREATE INDEX idx_requests_assigned ON requests(assigned_to);
```

### 3.3 Заполнение справочников и тестовых данных

При первом запуске сервера автоматически выполняется seed-скрипт (`internal/seed/seed.go`), который заполняет:

- 4 роли (admin, dispatcher, technician, supervisor);
- 4 типа услуг (Интернет, Цифровое ТВ, Телефония, Видеонаблюдение);
- 5 статусов заявок (new, assigned, in_progress, completed, cancelled);
- 4 категории (Подключение, Ремонт, Консультация, Модернизация);
- 5 тестовых пользователей;
- 10 тестовых заявок с реалистичными данными;
- 10 адресов в г. Казань;
- История изменений для каждой заявки;
- Уведомления для техников, диспетчера и руководителя.

## ГЛАВА 4. РАЗРАБОТКА СЕРВЕРНОЙ ЧАСТИ И МОБИЛЬНОГО ПРИЛОЖЕНИЯ

### 4.1 Архитектура системы

Система построена по архитектуре «клиент-сервер» с REST API:

```
┌──────────────────┐     HTTP/JSON      ┌──────────────────┐     SQL      ┌──────────────┐
│  Flutter App     │ ◄──────────────► │  Go API Server   │ ◄─────────► │  PostgreSQL   │
│  (Mobile/Web)    │     REST API      │  (Gin + Ent)     │    ORM      │  Database     │
└──────────────────┘                   └──────────────────┘             └──────────────┘
```

**Серверная часть** обрабатывает HTTP-запросы, выполняет бизнес-логику, управляет данными через ORM и возвращает JSON-ответы.

**Мобильное приложение** отображает интерфейс, отправляет запросы к API и хранит токены авторизации.

### 4.2 Серверная часть (Go, Gin, Ent ORM)

Серверная часть реализована на языке Go и имеет следующую структуру:

```
server/
├── cmd/server/main.go          — точка входа
├── ent/schema/                  — 10 Ent-схем сущностей
├── internal/
│   ├── config/config.go         — конфигурация из .env
│   ├── middleware/
│   │   ├── auth.go              — JWT авторизация
│   │   └── rbac.go              — ролевой доступ
│   ├── handler/
│   │   ├── auth.go              — логин, логаут, refresh
│   │   ├── request.go           — CRUD заявок, история, CSV
│   │   ├── notification.go      — уведомления

```

**Ent ORM** используется для:
- Описания схемы данных в виде Go-структур;
- Автоматической миграции БД;
- Типобезопасных запросов к базе;
- Eager loading связанных сущностей.

**Gin** — высокопроизводительный HTTP-фреймворк для Go, обеспечивающий маршрутизацию, middleware, валидацию запросов и формирование JSON-ответов.

### 4.4 REST API

API организован по принципу REST с версионированием `/api/v1/`:

**Публичные эндпоинты:**
- `POST /api/v1/auth/login` — авторизация (логин + пароль → JWT)
- `POST /api/v1/auth/refresh` — обновление access-токена
- `GET /api/v1/references/*` — справочники (типы услуг, статусы, категории, роли)

**Авторизованные эндпоинты (все роли):**
- `GET /api/v1/requests` — список заявок (с фильтрацией и поиском)
- `GET /api/v1/requests/:id` — детали заявки
- `PUT /api/v1/requests/:id` — обновление заявки
- `GET /api/v1/requests/:id/history` — история изменений
- `GET /api/v1/notifications` — уведомления
- `PUT /api/v1/notifications/:id/read` — отметить как прочитанное
- `GET /api/v1/profile` — профиль

**Менеджеры заявок (dispatcher, supervisor, admin):**
- `POST /api/v1/requests` — создание заявки
- `DELETE /api/v1/requests/:id` — удаление заявки
- `GET /api/v1/export/csv` — экспорт в CSV

**Только администратор:**
- `GET/POST/PUT/DELETE /api/v1/admin/users` — управление пользователями

### 4.5 Аутентификация и авторизация

Система использует JWT (JSON Web Token) с двумя типами токенов:

- **Access Token** (30 минут) — передаётся в заголовке `Authorization: Bearer <token>`. Содержит user_id, login, role.
- **Refresh Token** (7 дней) — хранится в БД, используется для получения нового access-токена без повторного ввода пароля.

Пароли хешируются с использованием bcrypt. Ролевой доступ реализован через middleware, которое проверяет роль из JWT и сравнивает с требуемой для данного эндпоинта.

## ГЛАВА 6. ДОКУМЕНТАЦИЯ

### 6.1 Руководство по развёртыванию

**Требования:**
- Go 1.23+
- PostgreSQL 16
- Flutter 3.11+

**Шаги установки:**

1. Создать базу данных:
```sql
CREATE DATABASE rostelecom_requests;
```

2. Настроить подключение в файле `server/.env`:
```
DB_HOST=localhost
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=ваш_пароль
DB_NAME=rostelecom_requests
```

3. Запустить сервер:
```bash
cd server
go mod tidy
go generate ./ent
go run ./cmd/server/
```

4. Запустить мобильное приложение:
```bash
cd mobile_app
flutter pub get
flutter run
```

### 6.2 Руководство пользователя

**Экран входа.** При запуске приложения отображается экран авторизации. Необходимо ввести логин и пароль. Система определит роль пользователя и предоставит соответствующий доступ.

**Главный экран — Заявки.** После входа отображается список всех заявок (для диспетчера/руководителя) или только назначенных (для техника). Заявки имеют цветовую индикацию статуса. Доступны фильтрация по статусу и поиск по клиенту.

**Создание заявки.** Диспетчер или руководитель нажимает кнопку «+ Заявка», заполняет форму: заголовок, описание, тип услуги, категорию, данные клиента, адрес. Опционально можно сразу назначить техника.

**Детали заявки.** При нажатии на карточку заявки открывается экран с полной информацией: статус, описание, данные клиента, адрес, назначенный техник, история изменений. Доступны кнопки действий в зависимости от роли.

**Уведомления.** Вкладка «Уведомления» показывает список оповещений о назначениях и изменениях. Непрочитанные выделены цветом.

**Профиль.** Вкладка «Профиль» отображает информацию о текущем пользователе, его роль и кнопку выхода. Для администратора доступна ссылка на управление пользователями.
